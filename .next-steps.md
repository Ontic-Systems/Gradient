### Latest Fix: Enum Block Detection for Type Declarations

**Problem:** The parser treated `type Error:\n    NotFound(msg: String)` as a record type declaration instead of an enum, because it saw `type Name:` followed by newline and immediately assumed record syntax.

**Solution:** Added `is_enum_block_rhs()` lookahead function to detect when the indented block contains enum variants (uppercase identifiers with possible constructors) vs record fields (lowercase identifiers with type annotations).

**Functions Added:**
- `is_enum_block_rhs()` - Looks ahead past newline/indent to check if content looks like enum variants
- `parse_enum_block_from_type()` - Parses enum variants in block syntax from type declaration context

## Remaining Issue: Typed Expressions with Constructor Values

The main remaining blocker is parsing typed expressions with constructor values:

```gradient
let error = TestError:
    UndefinedVariable(name: name)
```

**Analysis:**
- `TestError: UndefinedVariable(...)` is being parsed as a typed expression
- The value part `UndefinedVariable(name: name)` is a constructor call with named fields
- The parser needs to recognize this as an expression, not a record literal

**Test Case:**
```bash
# This works (inline):
echo 'type T: V(n: Int)
fn test(): T: V(n: 1)' | gradient-compiler

# This doesn't (indented):
echo 'type T: V(n: Int)
fn test():
    T:
        V(n: 1)' | gradient-compiler
```

**Files to modify:**
- `compiler/src/parser/parser.rs` - `parse_primary_expr()` and related functions
- The expression parser needs to handle newlines before constructor calls
