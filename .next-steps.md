### Fixes Applied This Session

#### 1. **Type Application in Typed Expressions** - FIXED
**Problem:** `Option[FnDef]: None` was failing to parse because the parser wasn't handling type application (`Option[FnDef]`) in the typed expression path.

**Solution:** Added lookahead for `LBracket` after identifiers in `parse_primary_expr()` to detect type application and handle it appropriately.

**Code Changes:**
```rust
// In parse_primary_expr() - added lookahead for type application
TokenKind::Ident(name) => {
    // Look ahead for type application: Option[Type] or Option[Type]: value
    if matches!(self.peek_ahead(1), TokenKind::LBracket) {
        let type_expr_spanned = self.parse_type_expr();
        let type_expr = type_expr_spanned.node; // Extract inner TypeExpr
        if matches!(self.peek(), TokenKind::Colon) {
            self.advance(); // consume ':'
            let value = self.parse_expr();
            // Return TypedExpr
        }
        // Handle non-typed case
    }
    // ... rest of handling
}
```

#### 2. **Constructor-Style Values in Record Literals** - FIXED
**Problem:** `TypeError: UndefinedVariable(name: name)` was being treated as a record literal field assignment instead of a typed expression with a constructor value.

**Solution:** Added check for `LParen` after field name in `peek_record_literal_field()` to detect constructor calls and reject them as record literal candidates.

**Code Changes:**
```rust
// In peek_record_literal_field() - added LParen check
TokenKind::LParen => {
    return false; // Constructor call, not record literal
}
```

#### 3. **Keyword Field Names in Record Literals** - FIXED
**Problem:** `consumed: Bool` in record declarations was failing because `consumed` is a keyword but not recognized as a valid field name.

**Solution:** Added `TokenKind::Consumed` handling to `parse_record_field_name()` function.

**Code Changes:**
```rust
TokenKind::Consumed => {
    self.advance();
    String::from("consumed")
}
```

### Immediate Priority: Fix Constructor Syntax in Typed Expressions

The main remaining issue appears to be parsing constructor calls with named fields within typed expressions. The parser needs to recognize that:


Is a **typed expression** with a constructor call value, NOT a record literal.
